Intelligence (Cross-Flow Analysis)

The Intelligence view is Hugin’s cross-flow aggregation engine. Where Nerve classifies parameters per-flow and the Sitemap groups by host/path, Intelligence aggregates across the entire project: every parameter ever seen, every reflection ever observed, every security signal ever raised, all rolled up by route and surfaced as actionable lists.

Use it to answer questions like:

  • “Which parameters in this project look like IDOR candidates?”
  • “Which endpoints reflect user input unencoded?”
  • “Which routes have stack traces leaking?”
  • “What’s my coverage on *.api.example.com?”

🔗Tabs

The Intelligence view splits into seven tabs.

🔗Params

Every parameter Hugin has ever seen across the project, deduplicated by name + context. Per-row stats: occurrence count, distinct value count, distinct endpoint count, value type (numeric, UUID, JWT, email, etc.), Nerve category if classified.

Sort by occurrence to find the most-trafficked params. Click any param to see all endpoints it appears on.

🔗Routes

All endpoints normalised to route templates (/users/42 and /users/43 collapse to /users/{id}). Per-route: methods observed, total flows, unique parameters, status code distribution, p95 latency, in-scope flag.

Click any route for the full per-route detail (params, reflections, recent flows, findings).

🔗Reflections

Parameters whose value reflects in the response body. Each reflection row shows: param name, route, encoding (none / HTML-entity / URL-encoded / JS-string), context (HTML body, attribute, JS string, etc.), confidence.

The Candidates sub-tab filters to unencoded reflections only — these are your XSS leads. Click any candidate to see the exact byte offset of the reflection in the response.

🔗Endpoints

Endpoints classified by type: api, auth, search, upload, admin, redirect, webhook, user, payment, integration, dev_staging, swagger. Each endpoint shows technology stack detected (Next.js, GraphQL, Spring, Django, etc.) and risk-relevant tags.

The detail view shows the full request/response for one example flow, plus suggested attacks per endpoint type.

🔗Rollups (Security Issues by Route)

Cross-flow security signals aggregated per route. Includes:

  • Reflection count
  • Sensitive data leakage occurrences (API keys, tokens in body)
  • Stack trace exposures
  • Open redirect opportunities
  • Permissive CORS
  • Cookie security issues
  • Mass-assignment surface (admin-flag-style params accepted)

The Summary sub-tab gives a project-wide overview: top-20 routes by issue count.

The Host sub-tab gives the same view scoped to one host — useful for end-of-engagement posture reports.

🔗Nerve (Persisted Signals)

The persistent layer for Nerve findings. Per signal: param name, category (idor/sqli/ssrf/…), confidence, route, evidence flow ID, first/last seen.

Filter by host, category, confidence. Use this to triage Nerve findings without re-running analysis.

🔗Client-Side

Persisted postMessage handlers and DOM sinks discovered across captured JavaScript. Per finding: type (postMessage origin / innerHTML sink / eval sink / etc.), severity, evidence flow, source code snippet.

🔗Response Intel

Cross-flow extraction from response bodies and headers:

  • Secrets (API keys, AWS access keys, private keys, JWTs)
  • Internal URLs / private IPs
  • Header issues (missing security headers, permissive CORS)
  • JS configs leaking (Mapbox tokens, Stripe public keys, Google API keys)
  • Cookie problems (missing flags, predictable values)

Filter by host, finding type, severity, signal name.

🔗Gold (High-Value Combined)

The Gold tab combines:

  • Nerve findings with confidence ≥ High
  • Client-side findings with severity Critical or High
  • Response intel findings with severity Critical or High

This is the “what should I report” shortlist — usually 10–50 items even on a large project.

🔗Auth Diff

The Auth Diff action takes two flow tags (tag_a, tag_b) representing different authentication contexts (e.g., tag_a=admin, tag_b=guest) and diffs the parameter sets. Parameters present in admin but not guest are IDOR / privilege-escalation candidates.

Workflow: tag your auth-as-admin captures with admin, your auth-as-guest captures with guest, run Auth Diff. The output is a ranked list of params worth attacking.

🔗Backfill

If you start using Intelligence after capturing a backlog of flows, run Backfill (analyze action) to extract intel from every existing flow that predates the engine. Set force=true to re-extract everything.

🔗Purge

The Purge Intel action clears the derived intelligence cache for a host or project. It resets the intel_done flag on flows so a subsequent backfill re-extracts cleanly. Granular: pass tables as comma-separated list (nerve,client_side,response_intel,reflections,params,endpoints,rollups) or all.

🔗MCP

The intelligence MCP tool covers everything:

  • Params: params, param_search, param_stats, param_endpoint
  • Routes: routes, route_detail
  • Reflections: reflections, reflection_candidates, reflection_stats
  • Endpoints: endpoints, endpoint_detail, endpoint_stats
  • Rollups: rollups, rollup_summary, rollup_host
  • Nerve: nerve_findings, nerve_stats
  • Client-side: client_side, client_side_stats
  • Response: response_intel, response_intel_stats
  • Gold: gold
  • Backfill: analyze (set force=true to reprocess all)
  • Purge: purge_intel (with optional host/project_id/tables scoping)
  • Auth: auth_diff (requires tag_a/tag_b for flow tags)
